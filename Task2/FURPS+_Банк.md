| Код | Требования                         | Комментарий  |
|-----|------------------------------------|--------------|
| F   | Функциональные (Functionality)     |              |
|     | Новый клиент видит список доступных депозитов с актуальными ставками на сайте                               |              |
|     | Клиент может подать заявку на депозит, оставив номер телефона и Ф. И. О                               |              |
|     | Менеджер кол-центра изучает заявку и предлагает особые условия                               |              |
|     | Идентификация клиента в отделении для подтверждения документов                                |              |
|     | В интернет-банке клиент видит персонализированные ставки и может подавать заявки на открытие депозита с подтверждением через СМС-код                                |              |
|     | Клиент получает уведомления по СМС или push                               |              |
| U   | Удобство использования (Usability) |              |
|     | Интерфейс должен cледовать системе дизайна банка                              |              |
|     | Быстрый отклик на операции                              | Меньше 1 секунды            |
| R   | Надёжность (Reliability)           |              |
|     | Защита чувствительной информации при передаче данных                               |              |
|     | Все сервисы должны работать 24/7 и быть доступны в 99,9% случаев                               |              |
|     | Избежать прямой работы интернет-банка с API АБС для повышения надежности                             |              |
|     | Возможность переключения на резервный ЦОД в случае сбоя                              | Разработать Failover стратегию           |
| P   | Производительность (Performance)   |              |
|     | Оптимизация загрузки справочных данных из АБС                             | Не более 1 секунды             |
|     | Горизонтальное масштабирование и распределение запросов между серверами и ЦОДами                               | Не в MVP              |
|     | Вертикальное масштабирование                               | Из-за ограничений тех. стека            |
| S   | Поддерживаемость (Supportability)  |              |
|     | Разработка микросервисов по депозитам в рамках MVP                              | Первый шаг для распила монолита и миграции с легаси-стека             |
|     | Разработка документации для дальнейшего расширения системы                               | Важный фактор для дальнейшего распиал монолита АБС            |
|     | Использование существующих технологий (MS SQL, Oracle) для интеграции новых решений с имеющимися системами                              |              |
|     | Обучение сотрудников новым технологиям          | Например, Kafka            |
| +R  | + Ограничения (Restrictions)       |              |
|     | Защита данных при передаче в интернет-банке с помощью механизма шифрования                               | Банковская тайна, регуляторные риски            |
|     | Поддержка функционала SMS в ядре системы с минимальными доработками со стороны подрядчика                               | Риски Vendor-lock            |
